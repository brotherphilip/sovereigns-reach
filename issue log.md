## [049] Fog of war non-functional — VisibilitySystem writes to GameState.visibility but view reads player["fog_of_war"] (always empty) | Severity: Medium | Status: Resolved
GameState has two separate visibility structures: `var visibility: Dictionary = {}` (object-level, rebuilt by VisibilitySystem.recompute()) and `player["fog_of_war"]` (player-level dict, initialized as {} and never updated). MacroViewController reads `player.get("fog_of_war", {})` to determine which tiles are revealed — this always returns empty dict → entire map appears unrevealed/hidden. VisibilitySystem.recompute(self) was called every game-day and correctly populated `self.visibility` from buildings and unit positions, but this data never reached the player dict the view layer reads. EventBus.fog_of_war_updated signal was also never emitted.
Resolution: After VisibilitySystem.recompute(self), copy `visibility.duplicate()` to `players[0]["fog_of_war"]` and emit EventBus.fog_of_war_updated with the visible tile keys. Scene test: ALL_SCENES_OK.

## [048] Raw material check missing in unit recruitment — siege engines recruitable with 0 wood/iron | Severity: Medium | Status: Resolved
`_cmd_recruit_unit()` checked gold and armory equipment but never verified raw material availability before deducting. `UnitRegistry.has_equipment()` explicitly skips raw materials (wood, stone, iron, etc.) since it only checks the armory. Siege units (battering_ram: 30 wood + 10 iron, trebuchet: 80 wood + 40 stone + 30 iron) have exclusively raw material costs. If the player had 0 wood, the deduction `maxi(0, 0 - 30) = 0` was a silent no-op and the unit was created for free. Players could recruit unlimited siege engines with no raw materials.
Resolution: Added a pre-deduction loop that checks raw material availability (`player["resources"].get(item, 0) < needed`) for any cost_resources item not in the armory, returning false if insufficient. Moved raw_cost declaration before the check. Scene test: ALL_SCENES_OK.

## [047] Most military units permanently unrecruitable — leather_armor, plate_armor, crossbows have no source | Severity: High | Status: Resolved
UnitRegistry defines: pikeman (needs leather_armor), swordsman, halberdier, captain, crossbowman (all need plate_armor), crossbowman also needs crossbows. No building produces leather_armor, plate_armor, or crossbows. The market (MarketSystem.BASE_PRICES) sold only bows, pikes, and swords. Player armory starts at 0 for all items. With issue #046 now also fixed (equipment deducted from armory), these five unit types were permanently locked — has_equipment() would always return false. Over half the unit roster was inaccessible, making mid/late game military progression impossible.
Resolution: Added crossbows (20g), leather_armor (15g), and plate_armor (25g) to MarketSystem.BASE_PRICES. Players can now purchase armor and crossbows to unlock the full unit roster. Market routing correctly puts these in player["armory"] via _add_resource(). Scene test: ALL_SCENES_OK.

## [046] Unit recruitment equipment cost never deducted — deducts from resources but equipment lives in armory | Severity: High | Status: Resolved
`_cmd_recruit_unit()` deducts `cost_resources` from `player["resources"]`. However, manufactured equipment (bows, crossbows, pikes, swords, leather_armor, plate_armor) is stored in `player["armory"]`, not `player["resources"]`. `has_equipment()` correctly gates recruitment on armory availability, but the subsequent deduction loop (`if item in player.get("resources", {})`) always failed silently because bows/swords/etc. are not in resources. Effectively: players needed 1 bow in the armory to recruit any number of archers — the armory was never decremented. The equipment economy (fletcher, poleturner, blacksmith chains) had no sink. This is a critical economy exploit enabling unlimited unit production with minimal equipment.
Resolution: Changed deduction loop to check armory first (`player["armory"].has(item)`) and deduct from armory, falling back to resources for raw-material costs. Scene test: ALL_SCENES_OK.

## [045] Fletcher and poleturner produce weapons without consuming wood — missing from ResourceTick PRODUCTION_INPUTS | Severity: Medium | Status: Resolved
BuildingRegistry defines `"fletcher": {"consumes": {"wood": 1}, "produces": {"bows": 1}}` and `"poleturner": {"consumes": {"wood": 1}, "produces": {"pikes": 1}}`. ResourceTick.gd has a PRODUCTION_INPUTS dictionary that specifies what each building must consume to produce; it contained mill, bakery, brewery, blacksmith but was missing fletcher and poleturner. As a result, fletchers produced bows without using any wood; poleturners produced pikes without using any wood. The ResourceTick.tick_building() check guards output on input availability — but with no inputs defined, the check always passed (empty dict = no requirements = always ok). Players could run unlimited bow/pike production with zero wood cost.
Resolution: Added "fletcher": {"wood": 1} and "poleturner": {"wood": 1} to PRODUCTION_INPUTS. Production now correctly halts when wood is unavailable. Scene test: ALL_SCENES_OK.

## [039] Religion coverage 10× too weak — raw ratio (0–1) passed where scaled delta (0–10) expected | Severity: Low | Status: Resolved
