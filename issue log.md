## [026] Population count never displayed in HUD — "Population:" label in right panel is an orphan stub | Severity: Low | Status: Resolved
HUDNode._build_right_panel() adds static label text "Population:" at y=222 via `_add_label()` but discards the return value — no reference stored, never updated in `_refresh_right_panel()`. The player's population count is shown nowhere in the HUD. The top bar shows gold/wood/stone/iron/food/ale/day/weather/prestige but has no population figure. The right panel's "Population:" is a dead-end label with no value.
Resolution: Added `_pop_count_label` member var to HUDNode. `_build_right_panel()` now stores the label reference (`_pop_count_label = _add_label(...)`). `_refresh_right_panel()` now updates it each refresh cycle as "Pop: N" from `player["population"]`. Population count is now visible in the right panel.

## [025] WorldMapScene uses has_method("server_config") — always evaluates false, world always seeds from 42 | Severity: Low | Status: Resolved
WorldMapScene._init_and_build() line 34–36: `GameState.server_config.get("map_seed", 42) if GameState.has_method("server_config") else 42`. `server_config` is a Dictionary var, not a method — `has_method()` always returns false. World map generation always used seed 42 regardless of the actual map seed set by setup_world().
Resolution: Removed the has_method guard. Now reads `GameState.server_config.get("map_seed", 42)` directly.

## [024] BuildingRenderer/BuildingLayer use wrong field names — buildings always render as "empty" (no work animation, fire animation loop broken) | Severity: Medium | Status: Resolved
BuildingRenderer.get_visual_state() line 21 read `building.get("is_operational", false)` — BuildingState uses `is_active` (never writes `is_operational`). Default false meant `is_op` was always false → condition `if not is_op or workers == 0` at line 43 always true → all buildings returned `"state": "empty"` (dark tint, idle animation) regardless of workers assigned. No building ever showed "working" state/animation. Separately, BuildingLayer._on_tick() set `_has_fire` by checking `b.get("state", "") == "fire"` on raw GameState building dicts (which have no `"state"` field — that field is in the computed view-state dict, not the simulation dict). `_has_fire` was always false, breaking the per-frame fire animation redraw in `_process()`.
Resolution: BuildingRenderer.gd: changed `is_operational` → `is_active`, default `true`. BuildingLayer.gd: changed `b.get("state", "") == "fire"` → `b.get("is_on_fire", false)` to read the actual simulation field. Buildings with workers now show "working" state; burning buildings now trigger per-frame animation redraw.

## [023] Unit movement orders never executed — Pathfinder and advance_along_path never called from GameState | Severity: Low | Status: Open
UnitState.issue_move_order() sets unit["order"]="move" and target_x/y. UnitState.advance_along_path(unit, path) exists and takes a precomputed path. Pathfinder.find_path() implements A* pathfinding. But GameState.simulate_tick() never calls Pathfinder or advance_along_path for player units on any tick. Player units placed on the macro map stay at their spawn position indefinitely. The core siege mechanic works via timed assembly (not unit position), so this is a design gap rather than a crash — but ISSUE_MOVE_ORDER commands have no effect.

## [022] UnitRegistry.can_recruit() checks is_operational instead of is_active — all units unrecruitable | Severity: High | Status: Resolved
`UnitRegistry.can_recruit()` line 325 checks `b.get("is_operational", false)` to verify the required building is operational. BuildingState uses `"is_active"` (set to true on creation, false on demolish/destroy) — `"is_operational"` is never written anywhere. The default of `false` means the check always fails, so every unit type that has a `requires_building` field (all units: peasant requires village_hall, scout requires village_hall, monk requires church, all military units require barracks/armory) could never be recruited. The player's entire military system was permanently locked.
Resolution: Changed `b.get("is_operational", false)` to `b.get("is_active", true)` to match the actual BuildingState field name and its default. Units can now be recruited when the required building is placed and active.

## [021] first_edict milestone inner check uses player_id field that doesn't exist in edict entry | Severity: Medium | Status: Resolved
Iter 54 fixed MilestoneSystem.check() being passed the server-level `active_edicts` (always empty). But the inner check `e.get("player_id", -1) == pid` still failed because EdictSystem.activate() stores edict entries as `{id, activated_at, expires_at, type}` — no `player_id` field. Since iter 54's fix already ensures we're iterating the correct player's own edict array, checking player_id is both wrong and redundant. The milestone was unreachable even after the iter 54 partial fix.
Resolution: Changed the inner check to `e is Dictionary and e.has("id")` — any entry with an id field in the player's own active_edicts array is a valid edict activation. "first_edict" milestone now fires on the first day after the player activates any edict.

## [020] is_starving never set — PrestigeSystem starvation gate permanently bypassed | Severity: Medium | Status: Resolved
`PrestigeSystem.calculate_daily_prestige()` returns 0.0 if `player["is_starving"]` is true (famine stalls prestige generation). `is_starving` is only set by `FoodSystem.tick()`, which was never called from GameState. `ResourceTick.tick_food_consumption()` handles the actual food deduction but does not update the flag. So even players with zero food stock continued earning prestige each day — no gameplay consequence for running out of food.
Resolution: Added a starvation flag update in `GameState._tick_player_economy()` after `FoodSystem.apply_granary_cap()`: `player["is_starving"] = FoodSystem.get_total_food(player) <= 0 and player.get("population", 0) > 0`. This is evaluated after consumption and cap enforcement so it reflects actual end-of-day food state. Prestige now halts correctly during famine.

## [019] Capital never auto-upgrades — donations accumulate but capital_level stays 0 forever | Severity: Medium | Status: Resolved
`_cmd_donate_to_capital()` recorded donations to `shire["capital_donations"]` but never called `CapitalSystem.upgrade()`. Even after meeting all upgrade costs (e.g. 200 stone + 100 wood for level 1→2), `capital_level` remained 0. PrestigeSystem._capital_multiplier() reads `shire["capital_level"]` directly, so the +10% prestige multiplier per level was permanently 0. Additionally, MacroViewController.gd read `shire.get("level", 0)` (wrong key) instead of `"capital_level"`, so the capital level display on the macro map was always 0 regardless.
Resolution: Added auto-upgrade check in `_cmd_donate_to_capital()` after `record_donation()`: calls `CapitalSystem.can_upgrade(shire, world)` and on success calls `upgrade(shire, world)`. Capital now upgrades automatically when cumulative donations exceed the tier threshold. Fixed MacroViewController.gd key mismatch (`"level"` → `"capital_level"`). The +10–50% prestige multiplier from capital levels now correctly applies once players have donated enough resources.

## [018] Edict passive modifiers never applied — all edict effects are permanently inert | Severity: High | Status: Resolved
EdictSystem.get_active_modifiers() was defined and returned a merged modifier dict for all active edicts, but nothing outside EdictSystem.gd ever called it. All 20 edicts had `modifiers` dicts (food_production_bonus, tax_multiplier, market_sell_price_bonus, fire_risk_reduction, food_consumption_reduction, wall_repair_amount, popularity_delta, etc.) but the simulation never read or applied any of them. Players could spend edict_points to "activate" edicts that had zero gameplay effect.
Resolution: Wired the five highest-impact modifier keys across the simulation. ResourceTick.tick_building() now multiplies food outputs by (1 + food_production_bonus). ResourceTick.tick_food_consumption() now multiplies daily_demand by (1 - food_consumption_reduction). TaxSystem.calculate_daily_gold() now applies tax_multiplier before difficulty scaling. MarketSystem.sell() now multiplies sell price by (1 + market_sell_price_bonus). GameState fire ignition loop now reduces fire_risk by (1 - fire_risk_reduction) using EdictSystem.get_active_modifiers(). Also fixed two instant-effect gaps in _cmd_activate_edict(): wall_repair_amount now calls BuildingState.repair() on all player buildings; popularity_delta now applies for non-summon active edicts. Movement/training/storage/combat modifier keys deferred (no hookpoints established).

## [017] edict_points never regenerate — entire edict system permanently locked (always 0 points) | Severity: High | Status: Resolved
GDD §7.1.2: "Points are spent to draft Edicts. Points regenerate slowly. Caps increase with prestige." EdictSystem.can_activate() checks player["edict_points"] >= cost_points (costs range 2–6). GameState initializes edict_points=0 (line 143) but no code ever increments it. Even after a full game session all edicts show "Insufficient edict points." The entire Royal Edict system (20 edicts, GDD §7) was permanently inaccessible.
Resolution: Added daily edict_points regeneration in GameState._tick_player_economy() at the day boundary: +2 per day, capped at min(20, 10 + prestige//100). Players reach the first low-cost edict (cost 2) after 1 game-day. The cap rises with prestige from 10 to 20, matching GDD §7.1.2 "Caps increase with prestige."

## [016] MilestoneSystem.check() passed empty server-level active_edicts — "first_edict" milestone unreachable | Severity: Medium | Status: Resolved
GameState._tick_player_economy() calls MilestoneSystem.check(player, world, milestones, active_edicts) where active_edicts is the server-level var (GameState.active_edicts, always empty — EdictSystem stores edicts in player["active_edicts"] not the server array). ## [014] setup_world() re-seeds only weather RNG — disease and fire RNGs use wrong seed for non-default maps | Severity: Low | Status: Resolved
